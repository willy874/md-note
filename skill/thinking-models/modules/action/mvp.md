# MVP（Minimum Viable Product，最小可行產品）

## 核心概念

MVP 不是「功能少的爛版本」，而是「能用最小成本回答最關鍵問題」的版本。
由 Eric Ries 在《The Lean Startup》系統化：每個新事物一開始都是**一組未被驗證的假設**，MVP 的任務是把其中最危險（風險最高、最會讓整件事崩盤）的那個假設，用最便宜、最快的方式丟進真實世界測試。

關鍵不是「做了什麼」，而是「學到了什麼」——能學到東西的簡陋版本 > 功能完整但不會有人用的精美版。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 想做一個**新產品、新功能、新副業**，但還沒有人用過、市場反應不明
- 已經「想了三個月」「寫了很多企劃」但一個使用者都沒碰過
- 打算花大量時間資源做一件事，但**最關鍵的假設**還沒被驗證（使用者會不會付錢？真的有這個痛點？通路行不行？）
- 完美主義傾向，想一次做到位，卻遲遲無法推出

不適合時機：
- 已有產品 / 穩定流程，要做的是優化 → 用 PDCA
- 純執行性任務（沒有未知假設要驗證）→ 用 timeboxing 或 two-minute rule

## 如何應用

1. **列出所有假設**：寫下這件事成立需要哪些條件為真（例：使用者有這個痛點、願意付 X 元、通路 Y 能觸及他們⋯⋯）
2. **找出最危險的假設**：哪一個一旦錯了，整件事就不成立？那就是 MVP 要驗證的對象。
3. **設計最小驗證**：不是「小一號的成品」，而是**能回答那個假設的最便宜手段**。常見手法：
   - Landing page + 付款按鈕（驗證需求 / 付費意願）
   - 手動 concierge（你自己當後台，驗證流程）
   - Wizard of Oz（前端看起來自動，後端是人力）
   - 影片 demo（Dropbox 最早這樣做）
4. **設定判準**：事前寫下「什麼數字代表假設為真 / 為假」。沒判準的驗證會被自我安慰劫持。
5. **跑出來 → 看結果 → Pivot or Persevere**：堅持、轉向、或放棄。

## 案例

- **Zappos**：創辦人不先囤鞋，而是去鞋店拍照放網上，有人下單才去買。驗證「網路賣鞋」的需求。
- **Dropbox**：寫完程式太貴，先做 3 分鐘影片演示雲端同步，email 訂閱從 5,000 漲到 75,000，才確定值得投入工程。
- **副業寫作**：想寫付費電子報，先發 10 篇免費文章到 Threads / Medium，看打開率 / 留言 / 訂閱請求，再決定要不要開付費版。

## 常見誤用

- **把 MVP 當成「簡陋版」**：做出 bug 滿天飛、體驗超差的版本，然後說「沒人用就表示市場不要」——其實是體驗太爛讓人無法判斷。
- **驗證錯的假設**：花三個月做了「很酷的技術」卻沒驗證「有人要付錢」。技術風險 vs 市場風險要分清楚。
- **沒有判準就跑**：跑完之後各種解釋「其實結果還不錯啦」，失去驗證意義。
- **MVP 完成後想直接長成正式版**：MVP 的程式碼很多時候是要丟掉的；它的產出是**知識**，不是**資產**。

## 與其他模型的關係

- MVP 是「第一圈」；後續迭代屬於 **PDCA** 或 **OODA** 的範疇。
- 與 **兩分鐘法則** 精神相通：降低啟動門檻、先做起來再說。
- 面對不可逆決策時，MVP 讓你把「不可逆」切成一連串「可逆」小步。對應 decision 模組的「可逆 vs 不可逆」。
