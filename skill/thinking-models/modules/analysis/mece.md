# MECE 原則（Mutually Exclusive, Collectively Exhaustive）

## 核心概念

MECE 由 McKinsey 顧問 Barbara Minto 在《The Pyramid Principle》中系統化，是顧問界拆解問題的底層文法。兩個要求：

- **Mutually Exclusive（彼此獨立）**：子集合之間不重疊。一個元素只會落在其中一類，不會被雙重計算。
- **Collectively Exhaustive（互相窮盡）**：所有子集合加起來能覆蓋整個母集合。沒有漏掉的角落。

為什麼重要？人腦拆問題時天然會**重複 + 遺漏**：想得到的面向反覆講，沒想到的面向完全忽略。MECE 是一個**紀律**，強迫你回頭檢查「有沒有重？有沒有漏？」。

關鍵不在「MECE 這個詞」，而在**選對切分軸（axis）**。同一個問題可以有多種 MECE 切法：

- 營收 = 新客 + 既有客（客戶來源軸）
- 營收 = 線上 + 線下（通路軸）
- 營收 = A 產品 + B 產品 + C 產品（產品線軸）

每種都 MECE，但**能解釋的事不同**。軸選錯，MECE 也沒意義。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 要做**結構化分析**：顧問簡報、策略報告、寫提案
- 使用者的拆解**有明顯重複或遺漏**：A 和 B 面向其實重疊、C 面向完全沒提
- 問題**範圍太大**，要切成可獨立處理的子題才能分工或逐個擊破
- 要設計**分類系統**：tag 體系、產品線、帳務科目、roadmap 分組

不適合時機：

- 問題本身就不該「窮盡」——例如發想創意時，MECE 會把思考壓縮到已知框架裡
- 面對**連續光譜 / 模糊邊界**的問題（美感、情緒、使用者意圖）——硬切會失真
- 快節奏溝通（日常回覆、聊天）——用 MECE 會變得太重

## 如何應用

1. **先定義「母集合」是什麼**：你要拆的到底是什麼？「提升營收」還是「提升新客營收」？範圍模糊時，MECE 沒法驗證。
2. **選切分軸**：常見軸有——時間、空間 / 通路、客戶類型、產品、流程階段、因果（原因 vs 結果）、質化 vs 量化。一次選一個。
3. **切出子集合**，然後做兩個檢查：
   - **獨立性**：一個元素會不會同時屬於兩個類？（例：「新客」和「線上客」會重疊 → 不能同軸並列）
   - **窮盡性**：有沒有漏掉的情境？加上「其他」可以救，但「其他」太大就表示軸選錯了。
4. **換軸再試一次**：同一個問題多換幾個軸拆，通常會冒出新洞見。軸的選擇本身就是 insight。
5. **往下遞迴**：每個子集合再用 MECE 拆一層，形成樹狀結構（Minto 所謂的 Pyramid）。

## 案例

- **營收下滑診斷**：營收 = 流量 × 轉換率 × 客單價。這是 MECE 切法（乘法結構，三者獨立且窮盡）。下滑時就能逐個檢查哪個變數崩了。
- **使用者分層**：活躍用戶 = 新用戶（首月內）+ 回流用戶（曾沉睡再活躍）+ 常態用戶（持續活躍）。按「首次活躍時間 + 當前狀態」軸切，不重不漏。
- **bug 分類**：bug = 功能錯誤 + 效能問題 + 體驗缺陷 + 相容性 + 安全。軸是「問題類型」。若混入「iOS bug」就違反 MECE（iOS bug 可能是功能也可能是相容）。
- **時間管理**：一天的工作 = 重要且緊急 + 重要不緊急 + 不重要緊急 + 不重要不緊急（Eisenhower 矩陣 = 兩個 MECE 軸交叉）。
- **簡報結構**：寫策略報告時，先列三個 top-level bucket，確認彼此獨立、合起來窮盡，每 bucket 再切三層。這就是 Pyramid Principle。

## 常見誤用

- **為 MECE 而 MECE**：硬要切得工整，結果失去真實世界的解析力。分類完了卻沒產生洞見，只是變整齊。
- **軸沒選對**：切法 MECE，但軸本身沒抓到問題關鍵。例：營收下滑時按「部門」切 MECE，但真正原因是「通路」——切錯軸看不到訊號。
- **混軸**：同一層混用兩個軸，必然產生重疊。例：客戶 = 大客戶 + 中小客戶 + 線上客戶（前兩個是「規模軸」，最後一個是「通路軸」，重疊）。
- **遺漏邊界 case**：把主要類別列完就結束，忘了 null / unknown / other。真實世界的資料往往 10–20% 落在邊緣。
- **把「窮盡」理解成「列越多越好」**：MECE 不追求顆粒度最細，追求的是「在當下這層足以決策」。過度拆解只是噪音。

## 與其他模型的關係

- **+ 第一性原理**：第一性原理幫你選對軸（把問題拆回事實）、MECE 幫你把拆解做得不重不漏。兩者搭配最強。
- **+ 魚骨圖**：魚骨圖天然就是 MECE 的視覺化——按類別展開可能原因。
- **+ 80/20**（prioritization 模組）：先用 MECE 拆解出所有子類，再用 80/20 找出真正重要的 20%。
- **+ 抽象階梯**：MECE 告訴你同一層要不重不漏；抽象階梯告訴你現在該站在哪一層。兩者互補：一個管水平，一個管垂直。
- **vs 5 Whys**：MECE 是「水平」展開（同一層列全）、5 Whys 是「垂直」深挖。互補使用。
