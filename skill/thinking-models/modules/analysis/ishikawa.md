# 魚骨圖 / 因果圖（Ishikawa Diagram）

## 核心概念

魚骨圖由日本品管之父 **石川馨（Kaoru Ishikawa）** 在 1960 年代為川崎造船所開發，原本是給基層作業員分析品質問題用的視覺工具。因為形狀像魚骨——中間一條主幹（問題），分支出去是「大骨」（主要類別），每根大骨再分出「小骨」（具體原因）——所以得名。

核心洞見：**多數複雜問題不是單一原因造成，而是多面向因素的交疊**。人天然會鎖定第一個想到的原因，魚骨圖強迫你**用預設類別當提示**，把可能原因全部展開，再逐一檢驗。

經典分類框架：

- **4M**（製造業）：Man（人）、Machine（機器）、Material（料）、Method（法）
- **5M**：4M + Measurement（量測）
- **6M**：5M + Mother Nature（環境）
- **4P**（服務業）：People、Process、Policy、Place
- **4S**（軟體）：Staff、Systems、Suppliers、Surroundings

類別不是聖經——可以依情境自訂，只要能**提示思考面向、不重不漏**即可（這裡和 [MECE](mece.md) 相通）。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 一個結果背後**可能有多個原因**，需要在追查前先**窮盡可能性**
- **品質問題、事故回顧、複雜故障** 涉及多個面向（人、流程、設備、環境）
- 要和**團隊一起腦力激盪**，需要有結構的框架避免亂槍打鳥
- 使用者**過早鎖定單一原因**（「就是他做錯的」「一定是設備問題」），忽略其他面向
- 要做**事前風險盤點**（launch 前列出可能出錯的原因）

不適合時機：

- 原因明顯只有一條因果線 → 用 [5 Whys](five-whys.md) 深挖
- 問題本質是**系統回饋循環** → 用[系統思考](systems-thinking.md)
- 單人思考、時間壓力大——魚骨圖的價值在**展開全景**，沒時間展開就用不上

## 如何應用

1. **明確定義「魚頭」（問題）**：可量化、具體。「產品品質差」不行，「第三季客訴中 62% 是包裝破損」才行。
2. **選一組類別當「大骨」**：依領域選 4M / 4P / 4S 或自訂。3–6 個類別最順手，太多會分散、太少會漏。
3. **每根大骨上腦力激盪具體原因（小骨）**：不急著評估，先窮盡列出。遵守 brainstorming 原則：不批評、先擴散。
4. **每個小骨可以繼續細分**：必要時對某個懷疑度高的原因，在那根小骨上繼續細分（這時會接近 [5 Whys](five-whys.md) 的用法）。
5. **用證據評估每根骨的可能性**：列完後，對每個可能原因評估——證據支持度、發生機率、影響大小。圈出最可疑的 2–3 個。
6. **對最可疑那條深挖**：用 [5 Whys](five-whys.md) 追根因，或用實驗 / 資料驗證。
7. **修正後回來檢視全圖**：問題解決了嗎？其他骨上是否還有隱藏風險？

## 案例

- **製造業經典**：產品良率下降
  - 人：新進作業員未完訓、夜班精神不濟
  - 機：設備校準超期、某批治具磨耗
  - 料：供應商換批次、濕度超標
  - 法：SOP 未更新、新參數未傳達
  - 量：檢測儀器漂移、取樣偏差
  - 環：季節性溫濕度變化、廠房改裝後粉塵

- **軟體服務可靠性**：線上活動當天頻繁 500 錯誤
  - Staff：oncall 排班遇假期、新人第一次值班
  - Systems：DB 連線池偏小、某 micro-service 記憶體漏
  - Suppliers：雲端供應商當天有區域性降級
  - Surroundings：活動流量是平日 20 倍、前端某 CDN 節點出狀況

  展開後才發現，單看 log 只會指向 DB，其實上游還疊加了流量暴衝與供應商降級。

- **行銷 campaign 成效差**：
  - People：創意團隊頻換、業主決策者換人
  - Process：A/B 測試沒跑完就全量
  - Policy：平台廣告政策改、某素材被降權
  - Place：主要投放通路觸及率下滑

- **自我反省用**：為什麼這個月進度落後？依「人（精神 / 學習）/ 事（優先序 / 任務設計）/ 時（時段 / 干擾）/ 地（環境 / 工具）」各列 3 個，通常會發現不是單一原因。

## 常見誤用

- **列完就結束**：畫了滿滿一張魚骨圖貼牆上，卻沒進入驗證與行動階段。魚骨圖只是**假設產生器**，不是答案。
- **把「不可能」的原因都列上**：列太多低價值選項，稀釋了真正重要原因的注意力。要有粗略的可能性篩選。
- **類別選錯**：4M/4P 不是聖經。若問題本質是「組織動力」，硬套「機器、料」只是湊字數。該自訂類別就自訂。
- **只做不寫**：魚骨圖的價值有一半在**視覺化**——讓團隊看見全景。只在腦中跑會漏掉面向。
- **和 [5 Whys](five-whys.md) 混淆**：魚骨圖是**廣掃**（同層多個可能），5 Whys 是**深挖**（沿一條因果追下去）。兩者不衝突但用途不同。
- **對單線問題硬用魚骨圖**：如果原因明顯只有一條，直接 [5 Whys](five-whys.md) 更快。

## 與其他模型的關係

- **+ [5 Whys](five-whys.md)**：魚骨圖列出「所有可能」，5 Whys 對最可疑那根骨深挖。先廣後深，是事故分析的標準組合。
- **+ [MECE](mece.md)**：魚骨圖的類別選擇本身要符合 MECE（彼此獨立、互相窮盡），否則原因會重疊漏列。
- **+ [系統思考](systems-thinking.md)**：若在魚骨圖上看到多條骨之間有**互相影響 / 回饋**（例：新進人員多 → 錯誤率高 → 老員工加班 → 離職率高 → 新進更多），要切換到系統思考。
- **+ 團隊工具**：魚骨圖天生適合**白板 / 共筆腦力激盪**。個人獨思效果較差，拉 2–3 人一起畫常常收穫最大。
- **+ [action 模組](../action/index.md) [PDCA](../action/pdca.md)**：圈出最可疑原因後，PDCA 負責設實驗驗證並持續改善。
