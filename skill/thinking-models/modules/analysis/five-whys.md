# 5 Whys（五個為什麼）

## 核心概念

5 Whys 由豐田佐吉（Sakichi Toyoda）提出、大野耐一（Taiichi Ohno）在 Toyota 生產系統中大量使用。原則極簡：**連續追問「為什麼」，每次的答案都再問一次為什麼**，通常到第 4–5 層就會碰到**結構性**或**流程性**的根本原因，而不只是表面症狀。

為什麼是 5 次？數字不是硬規定，而是一個**提醒**：人天生傾向停在第一個「合理的答案」，但那個答案通常只是**症狀的較深一層**，不是根因。5 次是經驗上「逼自己繼續問」的門檻。

根本洞見：**表面症狀治不了根**。如果只處理第一層答案（「他沒注意」「當天網路不穩」），下次類似問題還會再發生。真正的修正往往在第 4–5 層的流程、系統、設計缺陷。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- **故障、事故、bug、客訴**發生，要找根本原因而不是止血
- **流程重複出問題**（同樣的 bug 每季都來一次、同樣的客訴每月都發生）——表示上次沒挖到根
- 使用者停在**人為疏失層**（「他沒注意」「團隊不夠認真」）——通常後面還有流程 / 工具 / 培訓的空缺
- 想理解**使用者行為**（為什麼他們會這樣做？）深挖動機
- 做事故復盤（post-mortem）、retrospective

不適合時機：

- 問題**本來就顯而易見**（手滑打錯一個字）——硬追 5 次只是虛張聲勢
- 原因是**多因素並行**而非線性遞迴 → 用魚骨圖（廣掃）而不是 5 Whys（深挖）
- 涉及**系統回饋循環** → 用系統思考（看整體）而不是單線追問
- 使用者已經很清楚根因了，只是需要行動方案

## 如何應用

1. **清楚定義「問題」是什麼**：可量化、具體、可觀察。「系統掛了」不夠，「今天 14:30 結帳服務 5 分鐘內回應超時」才夠。
2. **問第一個為什麼，答案要基於證據而非猜測**：有 log、數據、訪談支撐。沒有證據的答案是自我安慰。
3. **每一層的答案都要能被驗證**：用「如果這個原因成立，那就應該會看到 X」來交叉檢查。
4. **不要停在「人」**：「因為 OOO 沒按流程」是常見誤區。繼續問：為什麼流程沒被遵守？流程是否合理？是否有工具 / 培訓 / 誘因的問題？
5. **必要時分支**：某一層答案可能有兩個原因並存，這時分成兩條線各自深挖。
6. **到「可以動手修」的那層就停**：不必非湊滿 5。太深會進入「人性、社會、宇宙」這種無法行動的層級。

## 案例

- **Toyota 經典案例**（大野耐一原版）：
  1. 機器停了 → 為什麼？
  2. 保險絲燒了 → 為什麼？
  3. 軸承潤滑不足 → 為什麼？
  4. 潤滑幫浦沒抽上油 → 為什麼？
  5. 幫浦入口濾網堵住，沒排除金屬屑 → 為什麼？

  根因是「金屬屑污染」+「濾網缺乏定期清理流程」。若只修到第 1 層（換保險絲），很快會再停。

- **線上服務當機**：
  1. 結帳超時 → 為什麼？
  2. DB 查詢變慢 → 為什麼？
  3. 某張表沒有 index → 為什麼？
  4. 上週的 migration 漏掉 index → 為什麼？
  5. PR 檢查清單沒有「index 檢核」項目 → 根因：**流程缺口**，不是「那個工程師粗心」。

- **使用者流失**：
  1. 第 7 天流失率高 → 為什麼？
  2. 第一週沒完成核心動作 → 為什麼？
  3. onboarding 沒有引導到該動作 → 為什麼？
  4. onboarding 是三年前設計的，當時的核心動作和現在不同 → 根因：**onboarding 沒有隨產品演進**。

- **自我提問**：「我為什麼又拖延？」→「因為不知道從哪開始」→「因為任務切得太大」→「因為我逃避面對它」→「因為我不確定結果會不會好」。根因是「害怕失敗」，行動方案是降低啟動門檻（兩分鐘法則）+ 重新定義成功標準。

## 常見誤用

- **變成找戰犯**：每層都歸咎給人（「就是他做錯」「他們不夠專業」）。Toyota 的精神是**對事不對人**，要找的是流程、系統、設計的缺口。
- **只追一條線**：真實問題往往多因素並存。只挖一條線會漏掉其他原因。需要廣掃時改用魚骨圖。
- **答案沒有證據**：憑直覺往下推，越推越離題，最後得出一個「好像很有道理」但毫無數據支持的根因。
- **硬湊 5 層**：有時第 3 層就已經是可行動的根因，硬湊到 5 層只是形式主義。
- **停在「因為沒時間 / 沒預算 / 沒人力」**：這些幾乎是萬用答案，沒有提供 insight。要再問「為什麼沒時間？是事情太多還是優先序錯？」
- **追到「人性 / 宇宙 / 資本主義」**：太深、無法行動。停在你能改的那一層。

## 與其他模型的關係

- **vs 魚骨圖**：5 Whys 是**單線深挖**（找到一條因果線，往下鑽）；魚骨圖是**多線廣掃**（列出所有可能原因）。實務上常先魚骨圖列出可能，再對最可疑那條跑 5 Whys。
- **+ MECE**：若某層的「為什麼」可能有多個並行原因，用 MECE 先把原因列不重不漏，再逐個深挖。
- **+ 系統思考**：5 Whys 假設「線性因果」。若發現是回饋循環（A 影響 B，B 又影響 A），改用系統思考。
- **+ bias 模組**：每一層的答案都可能被**確認偏誤**、**基本歸因謬誤**（把系統問題歸咎個人）劫持。挖完後回頭檢查每層答案是否帶偏見。
- **+ action 模組 PDCA**：找到根因後，PDCA 負責把修正落地並驗證是否有效。
