# 抽象階梯（Ladder of Abstraction）

## 核心概念

抽象階梯由語意學家 **S. I. Hayakawa** 在《Language in Thought and Action》中提出。他把語言與思考看成一道梯子：

- **梯子頂端**：最抽象——「財富」「幸福」「成長」「品質」這類大詞。涵蓋力強，但無法直接操作。
- **梯子中段**：中等抽象——「健康」→「有氧運動」；「成長」→「技能」；「品質」→「可靠性」。
- **梯子底端**：最具體——「今天走了 8,247 步」「用 Rust 重寫 parser」「P99 延遲 47ms」。可操作、可驗證，但難以通用。

核心洞見：**任何概念都可以在梯子上不同層級被表述，但選錯層級會讓溝通 / 思考出大問題**。

- **全在頂端**（永遠喊口號）：「我們要追求卓越！」「要做最有價值的事！」——聽起來很對，但沒人知道明天該做什麼。
- **全在底端**（陷在細節）：每天改一個 bug，但說不清楚產品在幹嘛。努力但沒方向。
- **兩人站在不同層**：一方講「我們要提升使用者體驗」，另一方問「所以我具體要改哪個按鈕？」——兩人都對，但對不上。

好的思考者能**在梯子上自由上下移動**：發現自己卡在某一層時，主動爬上或下一階。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 使用者或討論對象**卡在某個抽象層級**：只會講願景、或只會講細節，無法上下連通
- 一段對話**雞同鴨講**——仔細看是因為兩人站在梯子不同階
- 寫作 / 簡報 / 教學時要在**具體案例**與**抽象原則**之間切換
- 設計**抽象介面 / API / 產品分類** 時，需要判斷「這層該多抽象」
- 討論**策略 vs 執行、願景 vs 任務**看不到對應關係
- 使用者的主張**太空泛**（無法驗證）或**太瑣碎**（看不出重點）

不適合時機：

- 單純的事實查詢、技術操作——抽象階梯是思考工具，不是日常工具
- 情緒交流——抽象階梯會讓對話變冷冰冰

## 如何應用

1. **畫出當下的梯子**：對某個主題，列出至少三層——頂層（大概念）、中層（可分類）、底層（可操作）。例：
   - 頂：「產品變強」
   - 中：「留存率提升」
   - 底：「第 7 天回訪率從 12% 到 20%」
2. **檢查自己現在站哪一階**：若你在頂層（「我們要變強」），你無法行動；若你在底層（「改這個按鈕」），你可能失去方向。
3. **缺什麼層就補什麼**：
   - 卡在頂層 → **爬下**：要什麼證據才算「變強」？具體數字？行為？
   - 卡在底層 → **爬上**：這個改動是為了哪個中層目標？哪個頂層價值？
4. **對齊對話雙方的梯階**：吵架時先確認兩人站哪。A 在頂（「我要有安全感」）、B 在底（「那我每天幾點回家」），兩人都對，但要先在中層（「關係穩定 = 什麼樣的行為模式」）對焦。
5. **寫作 / 簡報公式**：抽象概念 → 中層原則 → 具體案例 → 回到抽象。讀者需要上下切換才能真的懂。純抽象變說教，純具體變流水帳。
6. **API / 抽象設計時的取捨**：抽象層級越高，重用性越強但學習曲線越陡；越低越好懂但不通用。依使用者能力與場景選對層。

## 案例

- **OKR 的本質就是抽象階梯**：
  - Objective（中～高抽象）：「建立業界最信任的支付體驗」
  - Key Results（底層）：「客訴率 < 0.3%、P99 延遲 < 200ms、NPS > 60」
  - 兩層連起來，願景才能被執行。只有 O 或只有 KR 都半殘。
- **產品會議常見災難**：
  - PM：「這功能能增加使用者黏著度」（中層）
  - 工程：「具體是哪個事件？幾% 提升？」（底層）
  - 老闆：「所以對公司策略有什麼幫助？」（頂層）
  - 三人都站在梯子上，但沒連通。好的 PM 會主動把三層串起來。
- **教學**：教「遞迴」時——
  - 頂：「用自身定義自身」（抽象）
  - 中：「分解問題為同型子問題 + base case」（原則）
  - 底：「斐波那契、河內塔、樹遍歷」（具體）
  - 只講頂看不懂；只講底只會背某題；三層都給才真的學會。
- **感情溝通**：伴侶說「我覺得你不在乎我」（頂）。有效對話需要爬下去——「不在乎」是什麼行為？（中）→ 「我昨晚跟你說事情你在滑手機」（底）。不爬下來就只能吵抽象仗。
- **寫作（Naval 風 / 高密度語錄）**：
  - 頂：「財富 ≠ 金錢」
  - 中：「財富 = 睡覺時還在幫你賺錢的資產」
  - 底：「程式碼、媒體、股權」
  - 三層合在一起讀起來才有力；缺哪一層就變陳腔濫調。
- **API 設計**：
  - 太抽象：`do(action, target, options)` 一個函式管一切——看起來通用，但沒人知道怎麼用
  - 太具體：每個動作一個 method，一百個 method——好懂但難維護
  - 中層：分類 resources（user, order, payment），每個有標準動作（CRUD），這是多數 REST 的甜蜜點

## 常見誤用

- **一輩子只會站一階**：
  - 只會頂（喊口號型）：「追求卓越」「以客為尊」，聽起來很對，沒人能執行
  - 只會底（埋頭苦幹型）：每天改 bug，但講不出產品在幹嘛
  - 解法是**刻意練習切換**：每次講抽象就自問「例子呢？」每次講具體就自問「這是為了什麼？」
- **同一句話不自覺跳層**：「我們要更有效率」（頂）「所以下週六要加班」（底）——中間的邏輯（哪個流程拖累 → 哪個環節優化）被跳過，結論就不可信。
- **逼對方爬到你的階**：你在底層問細節，對方在頂層想策略。不是誰錯，是要有人主動橋接。
- **以為抽象就是有深度**：把話講得玄才顯得高明。但如果沒有中層和底層連通，那只是空話。Naval 式的「高密度語錄」之所以有力，是因為他在底下早把階梯走完了，只拿最濃的一口給你。
- **爬得太高失去可驗證性**：抽到「幸福」「人生意義」這種層級後，任何主張都無法證偽，容易變成信仰而非思考。

## 與其他模型的關係

- **+ [MECE](mece.md)**：MECE 管「同一層」的切分不重不漏；抽象階梯管「層與層之間」的連通。兩者互補，一個水平一個垂直。
- **+ [第一性原理](first-principles.md)**：拆回事實時，要問「我現在拆到哪一層了？還能再拆嗎？」抽象階梯幫你量度深度。
- **+ [系統思考](systems-thinking.md)**：系統圖裡每個變數其實都有抽象層級，畫圖前先統一層級（不要一邊寫「幸福感」一邊寫「打卡次數」）。
- **+ [learning 模組](../learning/index.md)的[費曼](../learning/feynman.md)**：費曼技巧本質就是逼你在梯子上爬下到底層——能用具體例子講清楚，才是真懂。
- **+ 溝通 / 簡報**：好的敘事從抽象 → 具體 → 抽象反覆切換。純抽象變說教、純具體變流水帳。
- **+ [bias 模組](../bias/index.md)**：高抽象詞（自由、公平、成長）容易承載偏見。爬下到中層和底層，才能看出雙方真正在爭什麼。
