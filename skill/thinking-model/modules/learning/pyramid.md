# 由上而下溝通（Pyramid Principle）

## 核心概念

由 Barbara Minto 在麥肯錫任職期間系統化（著作 *The Minto Pyramid Principle*）。她觀察到顧問們資料多、寫得落落長，但客戶高層根本沒時間讀——於是把結構反過來：**結論在最上面，支持理由在中層，事實 / 數據在下層**，像金字塔一樣上窄下寬。

讀者從頂點（結論）決定「我要不要讀下去」——相信或好奇才往下看理由與證據。**主動權交還給讀者，不在寫作者。**

三條基本規則：
1. **一篇文章一個中心思想**。不要塞三件不同的事。
2. **每一層的點合起來 = 上一層的點**（MECE：彼此獨立、合起來無遺漏）。
3. **平行結構**：同一層的點要用相同格式、相同抽象層級（都在講「為什麼」、或都在講「怎麼做」，不要混）。

核心轉換：把「我想說什麼」換成「讀者需要聽到什麼、按什麼順序」。

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 使用者**寫完被問「所以結論是？」**——鋪陳太久、結論埋在尾巴
- 寫給**時間很少的讀者**（老闆、客戶、跨部門主管、評審）
- 郵件、Slack 訊息、週報、提案、投影片被嫌「太長看不完」
- 口頭報告、電梯簡報、面試開場，需要 30 秒講完重點

不適合時機：
- **說服性敘事 / 故事 / 品牌文案** → 反而應該鋪陳情緒、懸念先於結論
- **學術論文 / 探索性研究** → 格式有既定規範（但 Abstract 其實就是金字塔頂）

## 如何應用

1. **先寫一句話結論**（The Governing Thought）：如果讀者只能看一句，是哪一句？
2. **列出 2–5 個理由**支撐這句話——用「因為⋯⋯」去接。理由之間要 [MECE](../analysis/mece.md)（彼此獨立、合起來完整）。
3. **每個理由底下放 1–3 條事實 / 數據 / 案例**。這一層才是細節。
4. **檢查：倒過來讀還 make sense 嗎？**——去掉底層事實，只看結論 + 理由，邏輯應該仍成立。
5. **開頭用 SCQA 鋪情境**（Situation → Complication → Question → Answer），讓結論落地有脈絡，而非突兀跳出。
6. **寫郵件 / Slack 訊息**：第一行寫結論 + 想要的行動（「請 X 在 Y 之前確認 Z」），再補理由。

## 案例

- **週報**：
  - 傳統寫法：「本週我做了 A、B、C、D⋯⋯」（讀者看完還不知道重點）
  - 金字塔寫法：「本週最大進展是 X（結論），因為 1)⋯⋯ 2)⋯⋯ 3)⋯⋯。下週風險是 Y。」
- **Slack 求助**：
  - 傳統：「我在測 API，然後發現⋯⋯這個 case 有點奇怪⋯⋯你有空可以看一下嗎？」
  - 金字塔：「我卡住一個 API 500 錯誤，需要你協助檢查 log 權限（5 分鐘內可搞定）。詳情：⋯⋯」
- **提案簡報**：第一張投影片就把建議講完（「我們建議採方案 B」），後面每一張都在支撐這個結論。
- **面試自介**：先一句「我是做 X 的，過去 5 年在 Y 做出 Z」，再按金字塔鋪各段經歷。

## 常見誤用

- **把金字塔誤解成「只講結論不講理由」**：結論只是入口;下面的理由和證據仍要完整。
- **結論寫成「今天想跟你報告一下專案進度」**：這是標題不是結論。結論要是**判斷 / 主張 / 行動請求**（「我建議延後一週」「這個功能可以砍」）。
- **強迫所有場景都用**：故事、情緒訴求、懸念類寫作反而要鋪陳。用在該用的地方。
- **理由 MECE 失敗**：三個理由互相重疊、或漏掉一大塊——讀者會察覺但說不出來,只覺得「不太對勁」。

## 與其他模型的關係

- **+ [受眾思維](audience-first.md)**：金字塔提供「怎麼組織」,受眾思維決定「講哪個結論、為誰寫」。兩者必須一起用。
- **+ [費曼技巧](feynman.md)**：費曼幫你懂概念,金字塔幫你組織成讀者能快速吸收的形式。
- **vs 敘事結構**：兩者互補。高層報告用金字塔,品牌 / 募資 / 說服長文用敘事。知道何時切換。
- **+ [80/20](../prioritization/pareto.md)**（[prioritization 模組](../prioritization/index.md)）：金字塔頂那句結論就是 80/20 的具體體現——讀者若只能記一件事,就記那一件。
- **+ [MECE](../analysis/mece.md)**（[analysis 模組](../analysis/index.md)）：MECE 是金字塔每一層必須符合的結構紀律——彼此獨立、合起來無遺漏。
