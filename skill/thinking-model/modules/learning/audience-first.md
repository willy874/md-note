# 受眾思維（Audience First）

## 核心概念

**開始寫 / 講之前，先問三件事**：
1. **讀者 / 聽眾是誰？**——不是「所有人」，是某個具體的一群人，甚至一個具體的人。
2. **他們在意什麼？**——他們的痛點、目標、當下情境、既有認知。
3. **他們看完之後要做什麼？**——理解、被說服、做決策、採取行動、轉給別人？

多數溝通失敗不是因為內容不好，而是因為**作者從自己出發**（我想講什麼 → 我怎麼講），而不是**從讀者出發**（他需要什麼 → 我怎麼給）。

這是一個視角切換：從「我有什麼」變成「他需要什麼」。同樣的內容，換一個受眾就該用完全不同的講法——對 CEO 講 3 分鐘要結論，對工程師講 30 分鐘可以深挖原理，對家人講要去掉所有術語。

> Paul Graham：「寫作困難是因為你在用一種你自己也沒完全想清楚的方式，跟一個你根本沒想清楚是誰的人講話。」

## 使用時機

當使用者處於以下情境，載入此 submodule：

- 使用者**寫完沒人看 / 沒人回 / 沒人行動**
- 文章 / 簡報 / 郵件**技術度錯位**（對老闆寫工程細節、對工程師寫公關話術）
- 求職信、提案、貼文、影片寫了但轉化率低
- 跨部門溝通，對方總是「沒抓到重點」——多半是對方的重點和你不同
- 做行銷 / 內容 / 產品，對「我在為誰做？」答不清楚

不適合時機：
- **私人日記 / 純個人筆記** → 受眾就是未來的自己，不用過度設計
- **藝術創作的某些流派**（反受眾，表達自我為先）

## 如何應用

1. **動筆前先寫三行**：
   - **W**ho：讀者是誰（具體職位 / 角色 / 年齡 / 情境）
   - **W**hat：他們在意什麼（用他們會點頭的一句話描述）
   - **A**ction：看完要他們做什麼（一個明確動詞）
2. **想像一個具體的人**：不要寫給「開發者」，寫給「小明，30 歲後端工程師，剛接手一個遺留專案」。具體人讓你的筆有方向。
3. **檢查：有沒有用到受眾聽不懂的詞？**——每個術語都問「他會懂嗎？需不需要類比一下？」
4. **排版也是受眾的一部分**：老闆快讀用金字塔 + bullet，工程師深讀用章節 + code block，社群貼文要短句 + 強 hook。
5. **寫完讀一遍，問自己：「如果我是他，我看完會做什麼？」**——如果答不出來，就是沒有受眾思維。
6. **拿給一個真正的目標受眾測試**：回饋不是「寫得好不好」，是「你會做什麼 / 還有什麼不清楚」。

## 案例

- **技術部落格**：
  - 差：「本文介紹 Kubernetes 的 StatefulSet 使用⋯⋯」（讀者是誰？）
  - 好：「如果你正在把一個 Redis 叢集搬上 K8s，這篇告訴你為什麼 StatefulSet 能救你、以及我踩過的三個坑⋯⋯」（受眾 + 情境 + 承諾）
- **求職信**：不是「我是誰、我做過什麼」，而是「我理解你們正在做 X（展示你研究過公司），我有 Y 能幫上這個忙（對應到他們的需要），以下是證據（鏈結）」。
- **產品提案一魚三吃**：對 CEO 講「三個月 ROI / 市場風險」；對 PM 講「使用者流程、功能範圍、時程」；對工程 lead 講「架構影響、技術債、on-call 負擔」。同一件事三種剪法。
- **Slack 貼文 / 內部公告**：先想「80% 的人只會讀標題 + 第一句」——把最關鍵的結論 + 行動放在那。

## 常見誤用

- **「為所有人寫」= 為沒人寫**：想討好所有受眾,最後誰也沒打動。鎖定一個具體受眾反而能輻射更廣。
- **自戀式寫作**：通篇「我想講⋯⋯」「我覺得⋯⋯」,讀者視角完全缺席。這種文章自己讀很爽、對方讀很累。
- **把「受眾是誰」想得太抽象**：「開發者」「年輕人」「白領」——太大。具體到一個人的名字、角色、情境,才有方向。
- **誤解成「討好」**：受眾思維不是「告訴對方他想聽的」,而是「用對方能吸收的方式,告訴他該聽到的」。壞消息該講還是要講,只是用對方進得去的方式講。
- **做完才想受眾**：寫完一整篇才問「這要給誰看?」——此時結構已經歪了,重寫成本比從頭想大。

## 與其他模型的關係

- **+ [金字塔原則](pyramid.md)**：受眾思維決定「結論要講什麼、為誰講」,金字塔提供結構。兩者必須一起用。
- **+ [費曼](feynman.md) + [類比](analogy.md)**：一旦確定受眾是誰,費曼與類比才知道要「白話到什麼程度、用什麼比喻」。
- **+ [Steelman](steelman.md)**：Steelman 是受眾思維在「辯論」場景的極端形式——完全站到對方視角理解他。
- **+ [MVP](../action/mvp.md)**（[action 模組](../action/index.md)）：寫作 / 簡報 / 產品 launch 前,用 MVP 精神對一個小受眾測試,再放大——避免對著空氣寫三個月。
