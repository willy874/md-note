# 第 X 課 ESLint

說到各種專案基礎建置，eslint 可佔據舉足輕重的地位。它的功能是限制並控制程式碼的撰寫規範，透過程式的方式去檢驗團隊撰寫的規範，讓每個人程式碼的統一性達到一個基本的水平。

開始之前要稍微注意自己電腦的 nodejs 版本是否支援你要安裝的ESLint 版本，確認相符才是真的可以運行唷～

首先你可以先確認你想要用什麼樣的方式運行 eslint，他提供 JavaScript API、CLI、VSCode Extension ... 很多方式，個別情境都有合適的應用場景。但使用 ESLint 運行時的設定太過於複雜，我們需要寫一個 Config 給它進行設定調整。至於 Config 你可以選擇動態給予，但它背後運行時仍然會去照預設的 ESLint Config Path。不過其中使用 ESLint 最複雜的地方莫過於它極為變態的設定項目，也造就它驚人的生態系。

首先就是 Config 的命名有 eslintrc.js 外，也有新版的 eslint.config.js ，基於 nods.js 不斷更版，對於 ESModule 的支援越來越好， 現在都開始鼓勵使用 ESModule 來進行撰寫。但必須說新寫法截至撰寫這篇文章當下都還是支援不太好。

- 另外提及一下，自從 eslint@8.0.0 之後，官方不再去針對 formater 去支援，專注在程式碼結構的檢查。所以如果希望能檢查到格式化相關的規則還是要依靠 prettier 去進行格式化。但畢竟是不影響程式碼運行的規則，維護起來又頗麻煩，所以官方決定捨棄。

ESLint 是現在前端開發中很重要的工具，它能統一團隊程式碼的排版與寫作風格，透過工具的限制去提醒並強制開發者去統一風格。

但必須說，ESLint 其實非常複雜，要它背後運行原理與設定才能更好的去理解套件背後幫我們做了什麼事，這樣才能更有效去調校設定。
