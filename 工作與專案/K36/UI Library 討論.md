# UI Library 討論

這邊我提供當前的 UI Kit 和 Demo Page 給同事都看過了，並且 1V1 跟每個人聊過，確認大家想法。

相似意見概要：

- Paula, Otis, Sky, Vita 認為自己對 UI 相關沒有什麼開發經驗，所以不提供方向與想法
- Henry, Sunny, Jason 基本贊同這計畫，但也是沒有太多想法可以提供參考
- Plane, Yung 對開計畫抱持觀望Shawn 認為沒什麼必要

個人化意見：

- Sunny

- 提出使用 TypeScript 表示贊同

- Jason

- 使用 mui, antd 這類型的工具在後台會有不錯的收效，前台沒想法

- Vita

- 對於 Library CSS 方案，偏好 Tailwind ，但沒有具體執行想法。

- Shawn

- 覺得沒有什麼必要額外開發，認為當前元件使用是都可以處理問題的，大部分需求都可以透過 PM, BA 溝通協調一下或是前端同事微調一下就可以滿足需求。
- 主要是認為 PM 不太可能配合投入資源開發，所以態度抱持悲觀。
- 期待 BA 可以限縮情境，不要有太多特殊開發，可以讓 UI 開發一次到位。如果真的遇到特殊需求或是需要二次開發，偏好使用 mui, antd 這類型方案。
- 對於 Library CSS 方案, 以 CSS Module 為主力 ，Tailwind 不太適合。提出使用 TypeScript, Storybook 表示贊同。
- 特別要跟 UI Team 的使用習慣進行確認

- Yung

- 盡量沿用原本 nrc-formik，已 formik 相關功能為主。
- 認為 nrc-component 的淘汰與取代很有難度，可能方案再討論。
- 如果比較客製化則寫在主專案內加工就好，比如說 Modal 之類的元件。
- 如果真的遇到特殊需求或是需要二次開發，偏好使用類似 Redix UI, Base UI 這類型的客製化。相對不偏好 mui, antd 這類型方案，認為版本迭代很不好處理。
- 對於 Library CSS 方案, 以 CSS Module 為主力 ，Tailwind 不太適合。
- 各元件細節情況不一，可能都需要進一步分析討論。

- Plane

- 偏好使用 mui, antd 這類型方案，期待可以盡量選用符合規劃的 UI 庫去開發。
- 對於 Library CSS 方案, 以 CSS Module 為主力 ，Tailwind 嘗試空間。
- 不偏好用舊的，盡量去寫新的

個人想法：

技術想法

- 不管是 mui, antd 路線，或是 redix ui, base ui 路線，這兩條我覺得好壞參半，我都支持。
- CSS 方案我也認同使用 SCSS 就好，Tailwind 實在不太適合，有衝突舊 CSS 危險。
- 如果真的「很想要」使用 Tailwind ，可能要先嘗試在主專案導入，並且排除所有相關問題。

同意該計劃吃力不討好，也沒有辦法要求 PM 給予開發資源，最終大家時間花下去製作，導入成功率也不高。但我認為這件事是給大家一個比較實際的「平台」去發揮，可以讓大家有以下體驗：

- 了解元件開發
- 學習 TypeScript 使用
- 讓 Designer 和 Engineer之間深度溝通需求、嘗試寫測試
- 工作之間的一些空檔得以被運用，建立團隊的長期「支線任務」。
- 讓大家開始有想法上的刺激，而不是無情的需求執行機器。不避免會發生衝突，但大家會更有成長。

如果計畫是採用空檔開發，很可能根本沒有空，不是每個人都能夠每個 Spring 擠出時間。

最終計劃失敗我覺得也沒關係，就看可以推到哪。
