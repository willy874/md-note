# 型別

這是我針對打 API 底層方法進行 Type 包裝

PR 這個可以先參考

我覺得這是相對沒有侵入性的優化

但對於一些 API 常用功能更方便理解 Option

型別體驗通常是在 autocomplete 體驗到改善，有了靜態檢查後，降低也可以大幅降低 Rutime 不確定的錯誤發生。

型別複雜是正常的，如果要考慮維護性，隨時可以放棄型別，完全沒有侵略性。

如果要提高型別意識，可以先鼓勵採用 JSDoc，透過在檔案頂部加上 // @ts-check 語法單獨對檔案開啟型別檢查，不過可能需要教學 workshop 或執行低風險的重構進行前哨戰，未來比較有機會推動。

當專案有大量型別的上下文支持後，未來對於推動 AI 來加速寫 Code 會更有效。

當前我使用 AI 工具面臨的問題是上下文說明不清晰，導致幾乎無法 LLM 無法預測我的行為。

目前發現的問題：

專案內有大量死掉的 constants，不管是沒有使用，或是使用了但沒宣告。

底層操作中，仍然存在邊際問題，但目前確實不要動比較好。

還可以做的事：

可以針對 Option 強化說明文字部分

可以再進一步宣告「全部」Backend Service API 的型別，減少閱讀的不確定性，也可以加速反查前端屬性與 API 的關聯性。
