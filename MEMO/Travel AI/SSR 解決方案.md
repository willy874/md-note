# SSR 解決方案

以下是 SSR 解決方案，都是 React 能使用的框架

Next App Router

優點：

- 因為採用 Serverless，所以是 SSR 解決方案中擁有最快的處理速度
- 背後有大公司支持，使用這基數多，資訊也會快速增加
- 對於渲染場景的控制顆粒度細

缺點：

- 一定要採用 Serverless 的雲端服務。雖然可以捨棄 Serverless 架構，但就會失去其相關速度優勢。
- 架構相對複雜，掌握難度高

Next Page Router

優點：

- 有最悠久的歷史，有豐富的生態資源，熟悉的人也多
- 處理資料的心智模型單純

缺點：

- 目前核心開發團隊只維護，不提供新功能，不確定未來是否能成長

Astro

優點：

- 用來靜態渲染的框架，在 SSG 處理的開發者體驗很優良
- 不限制框架，可以跨框架相容

缺點：

- 當有過多動態邏輯和動態資料時會偏弱，需要混合 CSR 去解決頁面的問題，當遇到多頁面 CSR 還需要採用 Micro Frontend 架構
- 生態較小眾

Remix

優點：

- 輕量、好寫，心智模型容易學習。

缺點：

- 生態較小眾，有一派死忠支持者

TanStack Start

優點：

- 優良的開發體驗，整合多種前衛的開發技術

缺點：

- 目前社群呼聲雖然高，但官方還沒宣判正式版。

以下是靜態做法：

- 因為打包出來後是靜態檔案，很容易做 CDN，前端也不需要維護 Server

SSG

做法是每次打包時重新渲染 HTML，適合動態資料不多的網站

優點：

- 因為不需要渲染，擁有所有 SEO 解決方案中最快的速度

缺點：

- 因為是打包時渲染，使用者很有可能看到預渲染的資料
- 餵爬蟲時會有延遲的狀況，所以搜尋時會看到舊資料

Backend HTML Render

由後端在頁面產出時直接組裝 HTML，不考慮前端渲染結構，前端提供的渲染是純瀏覽器端的 CSR，產出的 HTML 不考慮畫面，只考慮爬蟲理解的語意

優點：

- 前端實作架構最簡單，可以直接做 CSR，

缺點：

- 後端多出SEO 專用頁面要維護
- 未來不確定搜尋引擎對該做法是否會出限制
