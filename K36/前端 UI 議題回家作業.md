請舉 1-2 個具體例子，說明你當時怎麼處理的。如果沒遇過也請說明。

- NrcSection

這是目前 Layout 使用範圍很大的元件，但因為參數的 title 沒有支援 ReactNode，Action button 許多 button 參數也都沒支援，也沒辦法加入其他節點，因而導致如果想額外加上 Icon 或 Tooltip 等等需求要捨棄整個Title Bar 的功能性。加上本身自帶 Pading，樣式實作時又要花時間把多出來的間距給消除。

- NrcModal

目前因為他必須要用 ref 去管理，所以對於狀態的資料流很凌亂，常常為了使用 ref 要多寫很多東西。目前後台已逐漸汰換成 BaseModal，BA 也幾乎以 BaseModal 為標準在設計。所以大多時候反而是直接捨棄，使用 BaseModal。

- NrcTable

有遇到每一個 column 的 tooltip id 是共用的，因而導致當觸發多個 tooltip 時，會造成 tooltip dom 座標標記錯誤。變成要自己實作外掛 Tooltip。

- NrcTabs

他會把未顯示的所有 Dom 都顯示出來，造成分頁多時會全部渲染出來，明顯影響到元件預期的生命週期和效能。

- InputField

強制一定要給 name，但某些邏輯並不希一望它受到 Form 的控制，等於強迫為了處理 name 的邏輯導至加工很多東西。

想要在特定節點追加 data attribute 很不好追加。

**### 承上題，扣分的原因是什麼？****###**

nrc-formik + yup 本身是沒什麼問題，只有欄位驗證的優化問題還需要去琢磨。

但最大的問題是在 nrc-components 上面，元件很多功能缺乏，對於普遍 Form Component 的實作上設計有相當大的缺失，也缺乏完整的文件、Use Case、規範，導致學習使用很困難，遇到問題又要回去爬程式碼。但反而 nrc-components 除了舊之外，很多設計邏輯過時且不夠適應現代開發需求，但其實大體功能都還算完整。

nrc-formik 雖然重新變成 Hook，但實作考量更加不全面，功能繼承也有很多缺漏，導致控制力缺乏與不易修改，這都是設計上的缺陷導致。

【向下兼容】為什麼你偏向這個選項？在什麼情況下你會改變這個選擇？\*

我會考慮未來可能要重構遷移，如果 API 落差很大會做起來很痛苦，要花很多時間去處理頁面和測試。但重新設計可以解決舊有參數的命名很亂的問題，讓參數隨著主流 UI Library 的風格，可以更好維護。

如果說放棄要重構遷移這個考慮，我會支持重新設計。

【Design Token】為什麼你偏向這個選項？在什麼情況下你會改變這個選擇？\*

因為 antd 的 Design Token 架構易用好懂，甚至我覺得比 Material Design 更容易理解。如果要自己重新定義，很可能會 Case 不完整，套用不齊全。如果不是要強調品牌調性的需求，我大概率都不支持自己重新定義語彙。多半我覺得不想用原有語彙的抗拒原因是學習，因為學好一門 Design System 的語彙不是一件短時間能掌控的事情。

Table】為什麼你偏向這個選項？在什麼情況下你會改變這個選擇？\*

Table 如果無法滿足需求時，客製化難度是超高的，但使用場景卻非常龐大。那如果是這樣，為何不一開始就以 Headless 的概念去設計，搭配 TanStack Table，在保持彈性的情況下做到最大的功能支援。

如果能最大程度的遵循 antd 的工能性，用原本的計有方案去滿足客戶需求，而非又去幻想一個 UI Component 的功能，那只使用 antd Table 似乎沒什麼問題。

【Form】為什麼你偏向這個選項？在什麼情況下你會改變這個選擇？\*

我覺得無論是資深或資淺，在公司也工作一年多了，大概也熟悉 Formik 使用。Formik 其實也跟 ReactHookForm 邏輯上差不多，優化議題也差不多，客製化概念也差不多。轉移技術的學習成本我認為是相對低的。

如果多數人都支持其他選項，其實我對於這個選擇沒有太大的意見。不管是 Formik, RHF 整合 antd form 還是使用純 antd form 我都覺得沒什麼問題。

【Form 邏輯歸屬】為什麼你偏向這個選項？在什麼情況下你會改變這個選擇？\*

我覺得 Form 的方案如果不是元件自成組合的功能（antd Form），而使用了 Formik, RHF 來進行整合後，如果又把它元件化，反而失去了它的彈性。

如果單獨開出 Field 元件在專案再整合讓 Form, Field, Input 溝通，不犧牲掉彈性的話，我就覺得支持放在 UI 庫。
