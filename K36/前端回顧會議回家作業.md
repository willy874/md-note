## i18n 與翻譯資訊管理段

**##**

**1.** **在本** **sprint** **中，前端要如何在開發前確認「是否需要處理** **i18n****」？**

請描述你實際會做的檢查步驟（例如：看哪些地方、問誰、檢查什麼）。\*

現況是跟 PM, BA 確認需要客製化翻譯的地方，會有一份 i18n 的對照試算表，或是直接寫在 Figma ，先針對裡面有定義的個別處理後，再去設計稿對照其他的文字欄位，用 AI 翻譯其他沒有處理的。整理出來後先放在開發的資料夾進入點統一整理，等到接近開發完成再一次整併到 locales/i18n。

但隨著需求處理和 Spec 變更，仍然可能發生變化。

**2. 當翻譯文件分散在 Jira / Figma 或只提供部分時，前端應採取的標準處理方式是什麼？**

請描述你認為可被團隊接受的一致作法。

現況：

目前是人工對照，或是 PM 把相關內容通通整合到試算表，或是 BA 整合到 Figma。

理想：

在開發前列出所有文字的翻譯列表，由專門的職務管理一份線上版本 i18n，總攬前後端所需要的翻譯，前後端開發人員可以用工具同步這份線上翻譯表。（因為有報表匯出功能，所以後端應該會有一份翻譯，但目前是前後不統一）

但困難點在這份工作會移出前端，會增加相關部位的工作量，並且需要相關人員具備基本的 html 知識。

**3.** **若前端沿用既有翻譯內容，最低限度應同步哪些人，並留下哪些紀錄？**

請具體說明紀錄位置與內容

目前通常是自己決定共用和沿用，不會跟任何人同步。

我的建議是「盡量不要有任何共用」，並形成標準。欄位以 Feature Scop 為前綴，以應用位置命名。專欄專用去應用在專案上，除了底層元件，不要有業務用翻譯共用。

缺點是翻譯量上升，但會減少維護的問題，也更容易追蹤廢棄的 i18n key API。

## 設計稿（Figma）正確性與變更回饋

**1.** **前端通常在什麼階段才會發現** **Figma** **與實作不一致？**

請描述實際經驗，而非理想狀態。

在處理優惠時，因為每一個步驟任一選項跟最終結果都有判斷的連動關係，但因為 BA, PM 其實沒有人熟悉完整的優惠建置邏輯，更沒有任何文件，所以 BA, PM, 一同在熟悉業務邏輯，看到有不合理和奇怪的才趕緊進行調整。而前端也是在實作 Spec 中特定功能時，基於原本的程式碼追查，如果發現有問題或邏輯對照不起來才會趕緊跟 PM 溝通。

**2.** **哪些類型的設計差異「一定要更新回** **Figma****」？哪些可以只留文字紀錄？**

請說明你的判斷依據。

我這邊所有邏輯對照不相符的都會請 BA 要更新回 Figma ，如此 QA 驗證時才能夠正確對照我預期的結果。

Jira 上的紀錄用來溝通，不作為長期的對照依據。

Code Comment 上的文字表述不容易理解的業務邏輯。

Figma 記錄完整的業務邏輯和情境。

**3. 當前端採用與設計稿不同的實作方式時，應如何同步 PM / BA 才不影響後續流程？**

基本上不允許，盡可能實作時要完全相符。不然就得回改到設計稿上，讓兩邊相符。

**4. 是否有必要建立「前端版 Figma 檢查清單」？若有，請列出你認為必須檢查的項目。**

可以規劃通用的檢查表和項目**，**如果是對於個別 Spring 的狀況跟情境可能就比較沒有效益，會吃掉很多 BA/PM 的工時，效益也不好。

但如果要推行更別需求的檢查項目，這件事可能是由 QA 來規範更加合適。

## 任務交接與責任透明度

**1.** **什麼情況下，前端應主動同步「任務負責人更換」？**

請描述具體情境。

如果是執行人更換但負責人不變，則不向往上的執行人額外同步。

比如說本次「站內信」情況是這樣：

專案負責：Lynn

任務BA：Yun

任務Reviewer：Vita

任務負責：我

執行人：我, Shopping

因為到後段有一些小 Bug 交由 Shopping處理，我只會跟 Lynn 報告有相關的問題依然找我。

所以 QA 開單會先開給我，由我消化理解後再轉交 Shopping，我不會跟 QA 報備執行人是誰。

如果有 UI 問題則 Shopping 自己找 Yun 溝通。

如果我休假則會告知 Lynn, Vita 進行工作調配，但負責人不變。

確實有點問題，點是 BA, QA 如果有狀況會以為變成是 Shopping 負責，但其實並沒有。

目前我覺得要在 Task 上面標注在 Frontends 把相關人員標上，或是描述開一個表格標記所有相關人員，以免不知道要找誰處理。

**2.** **前端在轉交或接手任務時，最低限度應補齊哪些資訊？**

（請至少列出 3 項）

- 目前進度
- Jira Tasks (其中已經包含 Figma 和 Spec，以及任務所有相關人員)
- Jira Tasks 上面沒有交代的資訊（reviewer, branch…)

**3. 這些資訊應該記錄在哪裡，才能避免只靠口頭或私訊？**

現況依然需要口頭和私訊進行討論與溝通，大量業務邏輯必須「實際看 Code 」才能知道業務細節。

我建議要有專門「文字版的Spec」，拔除 UI 行為和細節的操作，用來作為畫面設計與功能開發的元則基礎。

而任何需要用口頭說的情境應該要想辦法文字化或畫面化，而非靠看程式碼理解細節。

## Bug 處理節奏與狀態同步

##

**1.** **在什麼情況下，前端「一定要」更新** **Jira** **單上** **Bug** **的處理狀態？**

請描述具體時機。

這就必須先說當前的邏輯。

Toda：

- 未看到
- 已看到，但還沒開始處理

In progass：

- 已收到，但還沒開始處理
- 正在處理
- 已處理完成，SIT 測試中
- 已處理完成，等 Review
- 已處理完成，但有其他原因等待 Merge, Deploy

Ready To QA：

- 已處理完成並部署到QA 環境

Done：

- 通常是自己開的子單完成後

我大部分這幾個時間點去更新狀態，但很明顯會卡在 In progass 很久，也很難傳達進度，所已也常常要跟 PM, QA 同步說「為什麼那麼久」，並不是不處理，而是處理好卡在其他流程。

也很沒有統一定義「未看到」、「已看到，但還沒開始處理」、「已收到，但還沒開始處理」這些情境的切確狀況。

**2.** **一則讓** **QA** **能有效判斷進度的** **Bug** **狀態更新，至少需要包含哪些資訊？**\*

目前沒有切確的方案，只能看狀態讓 QA 知道能不能測，但實際進度沒有同步。

建議可以把上述情境細分成更細的狀態，但就要多很多狀態管理的成本，也很有可能狀態對齊上大家還不適應而不正確對齊實際狀態。

**3. 哪些 Bug 類型修好後應立即送測？哪些可以累積後再送？請說明判斷標準。**

大部分沒有嚴格定義，但目前只要是單獨的 Jira Task 都會獨立成單獨的 branch 去處理，會有獨立的單號個別管理。

但很可能累積一定數量後，統一時間才一起發 MR ，而後被一起 Review+Merge+Deploy。

**4.** **當前端工作量過高，影響** **Bug** **修復節奏時，應如何提早讓風險被看見？**

一種情況是 Bug 本身龐大：

本次站內信是到了 QA 階段的後期被開出「效能不佳」的 Bug，嚴格來說這是優化，但沒有在前期被注意到。因而作為 Bug 處理導致時程過於緊迫，產生差點無法再時限完成的風險。

一種情況是特定任務實際的工作量被錯估或延宕：

本次點數商城優惠設定這功能因為 API Spec 出來的晚，加上前端對該功能也相對陌生，導致更前期的設計處理不完整沒有盡早發現。也因這個任務龐大，又同時前端要處理同 Spring 的其它任務，任務量有點吃不消。另外事前沒有溝通到該 Feature 跟其他人的 Task 有依賴關係，需要等待該任務的完成，進而造成時限快到時才發現卡到所有人的任務節奏週期。

而這件事本身來看，對於任務定義是「Bug」還是「優化」還是「既有問題修復」本身就是很模糊的，會連帶其他議題產生。

而 Feature 之間有依賴關係時，如果還跨溝通問題，塞在同一個 Spring 完成本身就會非常緊迫。

每一個任務的依賴都有時間差，但如果是前端在等後端的時間差也不好處理，甚至是前端工作中也包含「對後端 API 的測試」這項工作，這是很難被估量的。

理想是前後端的開發應及被拆出來在不同的小時段被製作，前端功能之間的依賴更是，但當前執行策略上有困難。

## 工作量評估與延遲回報

**1.** **前端通常在什麼時間點才會意識到「需求可能做不完」？**

通常任務大小對于不熟悉真正業務細節的人不太能精準掌控，大多只能從「曾經開發過相似功能」去推算可能需要的時間，但很多時候開發病不是「全新的」，而是基於既有功能下去疊加做，所以對於工時實際需要的工時估計誤差非常大。通常製作到大概 50% 才會感受到真的做不完，但在這之前的不對勁會結論成「如果做不完再反應」，而不是基於「有點多，所以要調整工作量」，但這樣只要有任何特殊的修改、優化、需求依賴就會發生做不完。

**2.** **為什麼這個風險沒有被更早說出？請誠實描述實際原因。**

可以參照「Bug 處理節奏與狀態同步 (3.)」和「工作量評估與延遲回報 (1.)」

**3.** **前端可以用什麼方式呈現目前工作負載，讓調整能提早發生？**

通常發生這類型的超負載比較不是從進度上可以看到的，比較多會是「解壓縮」，能做的可能有：

- 先縮小要做的東西的範圍，過大的範圍會延遲開始實作有問題的區塊，進延遲發現
- 說明完整的業務邏輯，目前過度依賴「看 Code」來理解邏輯上下文
- 避免依賴別人的進度，因為同一個 Spring A 卡 B 就有可能發生沒辦法及早實作，導致延遲發現
- 無論怎麼做，仍然都有無法及早發現的風險，有不預期的問題出現現先上回報給上一層負責人，可以盡早調配人力或做專案調整。

以上如果能先做到，就可以盡早發現問題

## 跨部門技術溝通與紀錄」

##

**1. 前端在串接 API 時，最需要但最難取得的是哪些資訊？**

後端實際開出的結構其實都可能會跟預期落差很大，幾乎很難靠前端自己腦補就可以很相像。

具體舉幾個例子：

**案例一：**

今天有個物件有個含帶 ”name” 的屬性，可能原本預期得到 JSON 字串

“{ \\“en-US\\”: \\“Willy\\”, \\“zh-TW\\”: \\”威力\\“ }”

然而實際上卻得到 JSON 物件

`{ “en-US”: “Willy”, “zh-TW”: ”威力“ }`

如果沒有事先得到設計，這種只能瞎猜

案例二：

今天原本要拿到活動點數和點數效期，因為還不知道 Key，所以前端自己先設計結構。

`{

activePoints: number,

pointsValidityPeriod: number,

}`

因為舊有架構中不太能做一個命名轉換的抽象層，所以當得知正確的是：

`{

eventPoints: number,

pointExpiryDays: number,

}`

那就要大幅度改動很多檔案，把所有用到的地方和設定都要改到，如果刻意硬加抽象層反而是前端程式要大重構。

案例三：

很多屬性名稱很難一眼確認含義，甚至是會誤會，比方說：

vips

vipLevels

groups

原本看名字會以為是：

vipIds：VIP ID

vipLevels： VIP等級

groupIds：層級組別 ID

但事實上是：

vipIds：VIP等級ID

vipLevels： 層級組別 ID

groupIds：組別 ID

這如果不事前說明無法被預期

案例四：

API 資料儲存的邏輯和 UI 畫面並不相同。

當今天有一個業務需求是希望存下 VIP Level Range，使用者是選填寫 1~100 這樣的邏輯。

但實際上 API 卻要填寫切確 VIP等級 ID，也就是說 1~100 就要給予 1~100 精確那 100 個 ID。回傳時也是，要先確認欄位類型，然後轉成等級的數字排序驗證後再顯示給使用者。

**2. 若要由前端提供給後端的 API 範本，應該會是要長怎麼樣？(請提供具體範例)**

目前現有的做法是寫在 Jira 上，未來可能會提供 swagger 供參考。

最低標準要在需求開發前給予前端 swagger 相關文件。

需要針對每一支 API 的 url, method 定義，任何請求與回傳的 payload data schema 按照 OpenAPI 標準規範具體定義，而不是只提供「差異」。

如果徹底降低溝通成本要更完整：

- 有例外的 Error 情境需要標注回傳結構或是 Error Code。
- 有兩隻有依賴性的 API 需要具體說明。
- 有特殊用法的欄位，如(1.)上述多語系、VIP Level Range 等等特殊用法需要追加說明
- 後端阻擋的驗證邏輯（目前都是一律噴錯，無法在前端界定問題）

**3. 即時溝通（Gather / 口頭）後，前端最低限度應留下哪些文字紀錄？寫在哪？**

我這邊做法會在時間許可前提或架構許可前提針對開發過或溝通過的所有 API 細節在專案追加 JSDoc 補足遺漏資訊，以確保同事後續開發可以沿用。

但這麼做有維護風險，也不一定和後端是完全對齊。

最好還是由後端維護相對的文件，無論是 Confluence, Swagger, Google Doc 都會更好，再由前端基於文件撰寫 JSDoc。

**## 最後的問題**

**1. 以上六個面向中，哪兩項你認為「前端最有能力先改善」？為什麼？**

前端可以參與的改變：

「i18n 與翻譯資訊管理段」：可以逐步線上化，實作出線上管理的方案和流程，移出前端管理的範圍，減少一個層級的溝通成本。

「設計稿（Figma）正確性與變更回饋」：協助分析業務邏輯，讓邏輯不再依賴「口耳相傳」或「自己看程式碼」這樣的維護方式。

「任務交接與責任透明度」：可以協助把相關參與人員標注清楚，讓其他部門可以盡快找到協助處理問題的人員。

「Bug 處理節奏與狀態同步」：Jira 細化處理進度，讓 PM, QA 可以更快知道工作進度的細節。

「工作量評估與延遲回報」：前端立場當然希望盡早發現，但目前有點困難。只能仰賴對於業務邏輯熟悉的資深前端能不能先看出問題的端倪，或是及早意識分配到的工作量其實過於龐大。

「跨部門技術溝通與紀錄」：讓前端參與 API 的測試與驗證，就像 BA, PM 參與前端畫面的驗證一樣，及早解決問題。或是要有一套針對後端的 QA 機制，而不是只有測試前端畫面。

目前看最可行能夠被快速執行的是「任務交接與責任透明度」、「Bug 處理節奏與狀態同步」，其餘都需要制度、流程等等破壞性的更改。
