關於問題，有些事之前我就說要稍微提的

所以哪些是要呈報到跨團隊的還是要篩選提取一下

我這邊先整個寫出來最近整理的

**跨部門溝通問題**

1. 需求說「參考」某一頁的某某元件什麼功能，但希望某某地方要有什麼業務邏輯調整

但事實上這種「元件」是無法複用，甚至「元件」本身不是用「元件」的開發邏輯而是用需求業務邏輯在設計。位於工程師，這個時候的「參考」其實毫無意義，仍然需要情境完整的流程去歸納，而不是「參考」。「元件開發」和「需求開發」兩件事的差異，而不是長在一塊的東西就是元件，如果要參考等於是要花時間先全盤了解「參考」那個元件完整的業務邏輯與設計覆蓋情境，並基於覆蓋情境去修改和追加功能。

- 目前對於 BA, PM 如何辨識「元件」這件事比較強人所難，如果要追求共用和統一就必須建立元件的概念，否則應該每一次都清楚交代，否則永遠在開發「有歷史共業的全新業務元件」。

1. 需求分析

當同一個 Spring 中，有兩個需求有互相依賴和互相重疊到的模組，容易有相互修改相互覆蓋，甚至版本追回時難以追溯等等問題。在 Spring 之前要對工作切分不太會只是「需求的切分」，而是模組任務的切分，整個 Spring 下只有規劃者才知道模組分配。

- 某版開發一半要退版沒事先規劃到，變成不是 revert，而是「改回去」。
- 兩個人改到同一個模組時，要有人主導模組合併和管理。

2. 大數字的資料

團隊對於「龐大的 API 資料」並沒有很好的管理方式，大多是看到才處理，得制定一套解決方案。很多時候貪心要顯示更多資訊只會造成更多效能議題要處理。所以對於資料龐大的模組，如玩家、遊戲相關的頁面都容易遇上效能問題，需要更加高昂的開發成本。

- 資料過多的顯示方案（超過 1000筆，造成渲染效能瓶頸）
- 資料過多的效能瓶頸（超過 10000 筆，造成運算效能瓶頸）

3. API 介面

通常後端提供 API 介面都是有差異的地方，但實際串接時會動到的介面還會更多，甚至有上下文關係。每一個欄位的命名未必直覺，有時候反而要試過才會發現奇怪，然後進而去修改和定義。希望能有完整的 API 使用說明，減少閱讀理解難度和方便 AI 參考。

- 每次開發都能提供所需要的完整 API 介面描述，逐步補完文件，而不是永遠只給「新開發的」。
- 可以逐步補齊目前對於服務的上下文缺失

4. 前端程式邏輯的理解

- BA, PM 常常會有「顯示幾行」這樣的要求，但對於程式沒有「顯示幾行」這種概念，要做都是要用「算的」，多數時候不是 css line clamp 可以處理。
- Scroll bar 是有其特性限制，不是哪裡都可以長，也不是想長出來想消失都可以。
- 目前專案元件有一些既有限制，比方說:

- 下拉選單背後是 Position 做的，不是 Portal，這是要額外成本處理。如果不徹底解決會出現改 A 壞 B 的情境。
- SelectField 沒辦法隱藏過多的 Tag，所以目前內容過多都要找比較奇怪的解決方案

**前端問題：**

1. 元件設計

過去歷史上對於元件設計的成熟度不足，共用元件並沒辦法很好的覆蓋需求的成長。

- 元件混雜業務邏輯，難以抽離去共用或覆蓋合理場景。
- 元件樣式被過度封裝，要對齊設計搞的細節反而耗費更多時間調整樣式。
- 雖然有被 QA 提到元件行為各地方不統一，但也因為邏輯個別設計，新需求也沒有文件定義行為，更沒有一個公版做法，所以自然是不同。

2. 應該鼓勵大家寫 JSDoc

- 型別和功能說明可以幫助 AI 理解上下文，元件、API、工具函式等等
- 避免用過於動態的寫法，如果是固定的回傳欄位寫死寫未必是壞事，再另外轉換就好，這可以讓型別被更容易定義
- 寫文件很困難，但光只是把介面寫清楚對於下一個接手的已經是很好的上下文了

3. 既有的做法待討論

有寫法雖然是「行之有年」大家習慣的做法，但未必是好做法

- immutable 有可能過去很好用，所以專案深度綁定，但是會造成追查結果很困難。
- 過度依賴解構，而沒有去做好定義到底哪些要下傳，導致很難追查下傳的物件有什麼東西。
- 過度依賴 redux 暫存表單資料，redux 中間層做太多資料依賴轉換，難以追蹤資料流和資料生命週期。
- 噴錯不是壞事，不要為了不要噴錯而隱藏錯誤，補上無意義的 default value , try catch , error boundary，導致資料遺失和噴錯難以追蹤源頭。

1. 多國語系

多國語系因為是只有一隻檔案，可能需要按模組或 feature module 拆檔案。

- 會發生衝突，當開發到一半要 rebase 會衝突到很不好處理
- 檔案太大，不但會造成電腦卡，還很浪費 AI 的 Token
